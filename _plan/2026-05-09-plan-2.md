# Client-Side Verdict State Refresh Plan

## Summary

Replace submit-driven wizard state with session-state-driven inputs so the deal verdict is always derived from the latest client-side values. The wizard and every verdict-affecting expert page should recompute financing and verdict outputs on normal Streamlit reruns, while step navigation remains explicit and validation only gates moving forward, not saving data.

## Implementation Changes

- Remove `st.form(...)` from the deal verdict wizard steps and render plain widgets bound to `st.session_state` draft keys.
- Replace the current shared step-action helper with plain navigation buttons outside forms; no `st.button()` may remain inside a form, and any expert page that still needs a form must use only `st.form_submit_button()`.
- Keep the wizard’s draft keys as the input source of truth, but make them auto-persist on every edit. `Next` / `Back` only change steps and run per-step validation; they do not own data persistence.
- Add one shared recalculation entrypoint for the whole app, e.g. `refresh_deal_calculations(deal: DealProfile) -> None`, that:
  - normalizes financing fields from current purchase price and financing mode,
  - recomputes `verdict_outputs`,
  - backfills legacy underwriting fields from the verdict state when applicable.
- Call that refresh helper from:
  - the wizard after syncing draft state into `deal_profile`,
  - verdict-affecting expert pages after widget values change,
  - the summary/report page before rendering metrics.
- Add a compact live verdict preview component to the wizard’s non-final steps. It should render current status plus a small metric snapshot from current session state, and the final Verdict step should keep the full summary.
- Retrofit verdict-affecting expert pages that currently use forms (`market/property`, `acquisition`, `capital markets`, `underwriting` where relevant) to plain widgets so changes rerun immediately and update the shared derived verdict.

## Public Interfaces / Types

- Add `refresh_deal_calculations(deal: DealProfile) -> None` as the single shared recalculation API.
- Keep `evaluate_deal_verdict(deal: DealProfile) -> VerdictOutputs` pure and calculation-only.
- Keep wizard-local sync separate, e.g. `_sync_draft_to_profile(deal_profile) -> None`, so Streamlit state handling stays out of logic code.
- No new persisted model is needed; `DealProfile` remains the shared app state object and `st.session_state` remains the client-side input store.

## Test Plan

- Wizard regression: rendering any step no longer raises the `st.button() can't be used in an st.form()` error.
- Wizard live state: editing rent, purchase price, rate, or expenses updates derived verdict metrics without a submit action.
- Navigation validation: incomplete required fields still block forward movement, but entered values remain in session state.
- Cross-page sync: changing purchase price, financing, or underwriting-related inputs on expert pages updates the shared verdict/report state on rerun.
- Report consistency: `Deal Summary Report` and the final wizard step show the same verdict and reasons for the same deal state.
- Boundary coverage: existing Pass/Caution/Fail threshold tests still pass after refresh flow changes.

## Assumptions

- “Client-side state” means Streamlit `st.session_state`; no backend persistence or external storage is added.
- Whole-app scope includes all pages that change verdict inputs or dependent financing data, not unrelated pages like CRM.
- Explicit “Save” buttons are no longer required for verdict-affecting pages; edits persist in session state immediately.
